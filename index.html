Context

The reengineering and reverse engineering of software systems is gaining importance in software industry, because the accelerated turnover in software companies creates legacy systems in a shorter period of time. Usually, these legacy systems are critical the software company business, model. Our case study is over 3 000.000 lines of code commercial software written in PowerBuilder. The system is more than 20 years old and it's managed by a version control system. Because of the rising market demand, there is no team for system maintenance. As consequences developers burly improve or fix defects. Here we present our reengineering process. We use the model driving approach and analytics in Pharo Moose environment

Tickets database

A ticket is related do a task to do. This task can be fixing a bug, writing documentation, adding a new feature, etc. We start our process by analyzing the tickets database over time. A ticket has the following characteristics:

  1. the libraries it related to
  2. the creation date
  3. the closing date
  4. time spent by a developer
    1. time to analyze
    2. time to develop
    3. time to test
  5. the ticket it's related to
  6. time to test by quality team
  7. file it related to ( it can be found the source code comment)
  8. The module is related to.
To have a visual idea of the tickets database, we plot bugs tickets and evolution tickets in the same graphic for each PowerBuider library in the studied project. Here are some visualization samples

Visualization of bugs and evolutions tickets per libraries

Our observation across all libraries is that there are some libraries which have bugs tickets than evolutions cot_uo for example. Some of them have as many evolutions as bugs and others seem to not have bugs. This situation correlate with what developers actually think about the system.


Visualization of time on bugs and evolutions tickets

To evaluate the situation over time, we plot average time of development on tickets, time to open and close tickets, time to test tickets for bugs and evolutions from 2004 to 2019. To smooth the graph we use moving average with a scale of two months before and two months after. We plot a regression line for each graph to approximate its behavior.

As we can see developers tend to spend more and more time to make any modification in the system. However, testing team trend to less tested the system after modification.


Tickets analysis dashord 2020 output

The dashboard presents the time spent solving and testing tickets from 2004 until 2020. One can see a huge augmentation of development from 2019 to 2020, drop-in testing by the RC team and at a slight increase of testing time spent by developers team. This situation shows that regression tests aren't prioritized in 2019, and expose the system to regression bugs.

Toward Duplication removing and God class refactoring

After the first step of analysing the system situation. We realise that there is a reengineering need for the system to improve the situation. This said, we will first focus on a strategy to remove dupplication from the system, and decomposing god classes without regression.

God class refactoring Strategy

God classes are classes in charge of too many responsabilities in a software system. Many publications describe how to detect them . But removing or refactoring them is not easy task. It need to be first planned (Decomposing God Classes at Siemens Nicolas Anquetil et al.). This said we decided to focus on one god class it outgoing invocation and incoming invocation. We confronted this to the developer knowledge about the system .

This is a visualization of our system references between objects restricted to uo_taitement_liq_cal our god class.

Developer could directly see at a glance unused references or unused object. We decided with developpers to have a tagging system where we can tag an element depending on there expertise of the system.
